## 三、Bug 修复：密钥编辑弹窗

### 3.1 Bug 描述

点击「编辑」按钮后弹窗崩溃（React 白屏）或表单字段全部空白，无法提交。

### 3.2 根因（三层洋葱结构）

| 层 | 问题 | 原因 |
|:--:|------|------|
| 外层 | `api_id` 未填入 `initialValues` | 表单 `required: true` 但下拉框空白 |
| 中层 | `initialValues` 完全失效 | antd v5 + React 19 兼容问题，`destroyOnClose` 废弃 |
| 里层 | `form.setFieldsValue` 传 `new Date()` 导致 React 崩溃 | antd DatePicker 底层是 dayjs，不兼容原生 Date |

### 3.3 解决方案

```tsx
// 1. 导入 dayjs，用 useEffect + form.setFieldsValue 替代 initialValues
useEffect(() => {
  if (open && editKey) {
    form.setFieldsValue({
      api_id: Object.keys(editKey.access_rights)[0],
      expires_at: editKey.expires > 0 ? dayjs(editKey.expires * 1000) : undefined,
      // ...
    });
  }
}, [open, editKey, form]);

// 2. onFinish 编辑模式保留原 access_rights
if (editKey?.access_rights?.[values.api_id]) {
  payload.access_rights = { [values.api_id]: editKey.access_rights[values.api_id] };
}
```

### 3.4 教训

1. antd v5 + React 19 是已知不兼容组合，DatePicker 只认 dayjs
2. `useEffect + setFieldsValue` 比 `initialValues` 可靠
3. 浏览器手动点一下比自动化验证快 10 倍

## 四、架构演进：管理数据库

### 4.1 为什么需要数据库

Tyk OSS 仅用 Redis 存储当前生效的 API 定义。没有持久化、没有历史、没有审计。"已停用"其实就是 DELETE——删了就没了，想恢复只能重新手配。

管理数据库解决三个核心诉求：
- **配置持久化**：不随 Redis 清空丢失
- **自管状态**：实现 OSS 缺失的"启用/停用"
- **变更追溯**：谁、什么时候、改了什么

### 4.2 方案迭代

| 版本 | 方案 | 结果 |
|:--:|------|------|
| 1 | Supabase (PostgreSQL) | ❌ Supabase 存在问题 |
| 2 | Docker PostgreSQL | ❌ 网络阻断，docker pull 全线超时 |
| 3 | sql.js (SQLite/WASM) | ✅ 但纯浏览器存储，非真正独立数据库 |
| 4 | **PostgreSQL + PostgREST** | ✅ 最终方案：HTTP_PROXY 拉镜像，Docker 本地运行 |

### 4.3 最终架构

```
┌─────────────────────────────────────────────────────────┐
│ 浏览器 (localhost:5173)                                  │
│   ├── /tyk/* → Tyk Gateway (localhost:8080)             │
│   ├── /hello → Tyk Gateway                              │
│   └── /db/*  → PostgREST (localhost:3001) → PostgreSQL  │
│                                                         │
│ Refine DataProvider                                     │
│   ├── tykFetch() → Tyk REST API (主流程)                │
│   └── ichse-db.ts → fetch(/db/...) → PostgREST (双写)   │
└─────────────────────────────────────────────────────────┘
```

### 4.4 双写覆盖

| 操作 | Tyk | DB |
|------|:--:|:--:|
| 创建 API | POST /tyk/apis/ | INSERT api_definitions + INSERT log |
| 更新 API | PUT /tyk/apis/{id} | UPDATE api_definitions + INSERT log |
| 删除 API | DELETE /tyk/apis/{id} | UPDATE status='inactive' + INSERT log |
| 创建密钥 | POST /tyk/keys/create | INSERT api_keys |
| 吊销密钥 | DELETE /tyk/keys/{id} | UPDATE status='revoked' |

DB 操作失败不影响 Tyk 主流程（console.warn 降级）。
